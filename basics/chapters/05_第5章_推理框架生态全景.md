# 第5章：推理框架生态全景

## 5.1 通用推理引擎对比

### 5.1.1 TensorRT 详解

TensorRT 是 NVIDIA 推出的 GPU 专用推理引擎，是工业部署的事实标准。

核心特性：

算子融合：自动融合 Conv+BN+ReLU / MatMul+GeLU，减少 kernel 启动与 HBM 读写。

INT8/FP8 量化：内置 Entropy Calibrator，支持 PTQ 量化。

Kernel Auto-Tuning：根据 GPU 型号自动选最快 kernel。

Dynamic Shape：通过 Optimization Profile 支持可变输入维度。

支持模型：CNN（ResNet/YOLO）、Transformer（BERT/ViT）、LLM（需配合 TRT-LLM）。

典型工作流：PyTorch → ONNX → TensorRT Engine → 部署

优势：

性能极致，相对 PyTorch 原生推理 2-10 倍提升。

工业级稳定，全球头部公司都用。

劣势：

只支持 NVIDIA GPU。

版本兼容差（CUDA/TRT 版本绑定严格）。

自定义算子需写 Plugin（C++，门槛高）。

适用场景：云端 GPU 推理、Jetson 边缘部署。

### 5.1.2 ONNX Runtime 详解

ONNX Runtime 是微软推出的跨平台推理引擎，是 ONNX 标准的官方实现。

核心特性：

跨平台：Windows/Linux/macOS/Web，CPU/GPU/NPU 全覆盖。

Execution Provider（EP）：通过 EP 机制对接各硬件后端（CUDA/TensorRT/CoreML/DML）。

算子覆盖广：支持 ONNX 标准全部 opset（17-21）。

易用性：3 行 Python 即可推理。

典型代码：

```python
import onnxruntime as ort
sess = ort.InferenceSession("model.onnx", providers=['CUDAExecutionProvider'])
outputs = sess.run(None, {'input': x})
```

优势：

真正的跨平台（一套 ONNX 到处跑）。

易用性极佳，上手 5 分钟。

微软生态强大，文档完善。

劣势：

性能不如 TensorRT（通常慢 20-30%）。

复杂模型（如 LLM）优化深度不足。

适用场景：跨平台部署、原型验证、传统模型生产部署。

### 5.1.3 OpenVINO 详解

OpenVINO 是 Intel 推出的开源推理引擎，针对 Intel 硬件深度优化。

核心特性：

Intel CPU 深度优化：AVX-512 / AMX 指令集，CPU 推理性能领先。

集成 Plugin：也支持 NVIDIA GPU / ARM 等其他硬件。

Model Optimizer：从 TF/PyTorch 直接转 OpenVINO IR。

NNCF（Neural Network Compression Framework）：内置 INT8 量化工具。

优势：

Intel CPU 性能最强（适合服务器 / 工控机）。

完整工具链（量化 + 转换 + 推理 + 部署）。

开源免费。

劣势：

Intel 生态绑定（非 Intel 硬件性能一般）。

LLM 支持较弱（远不如 vLLM/TensorRT-LLM）。

适用场景：Intel CPU 服务器部署、工业视觉、智慧城市（大量 CPU 节点）。

## 5.2 LLM 专用框架对比

### 5.2.1 vLLM 详解

vLLM 是 UC Berkeley 2023 年推出的 LLM 专用推理引擎，目前是开源社区事实标准。

核心创新：

PagedAttention：借鉴 OS 虚拟内存，KV Cache 分页管理，显存利用率 20% → 90%。

Continuous Batching：动态拼装 batch，GPU 利用率 30% → 80%。

Prefix Cache：自动复用相同前缀的 KV（如 system prompt）。

多卡 TP/PP：内置 TensorParallel / PipelineParallel 支持。

典型用法（OpenAI 兼容 API）：

```bash
vllm serve <model_name> \
  --tensor-parallel-size 4 \
  --enable-prefix-caching \
  --max-model-len 32768
```

优势：

性能强劲（吞吐相对 TGI 提升 10-20 倍）。

OpenAI 兼容 API，业务接入零成本。

社区活跃，新模型/特性快速支持。

劣势：

仅支持 NVIDIA GPU + 部分 AMD/Intel。

二次开发复杂（C++/CUDA 经验）。

适用场景：通用 LLM 云服务、企业内部 LLM 平台、学术研究。

### 5.2.2 TRT-LLM 详解

TensorRT-LLM 是 NVIDIA 推出的 LLM 专用推理引擎，性能极致但使用门槛高。

核心特性：

In-flight Batching：NVIDIA 版本的 Continuous Batching。

PagedAttention：原生支持。

Plugin 体系：FlashAttention / GQA / MoE 等通过 Plugin 实现。

多精度支持：FP8（Hopper）/ INT4 SmoothQuant / FP4（Blackwell）。

多卡 TP+PP：完整支持。

典型工作流：

HuggingFace 模型 → build.py 转 Engine → run.py 推理

优势：

性能极致（比 vLLM 再快 20-50%，视场景）。

NVIDIA 官方支持，硬件新特性第一时间支持（FP8/FP4）。

Jetson 平台无缝部署。

劣势：

使用门槛高（编译模型耗时 10-30 分钟）。

只支持 NVIDIA GPU。

版本绑定严格（CUDA/TRT/Driver 三方绑定）。

适用场景：NVIDIA 集群极致性能、Jetson 边缘 LLM、商业大模型 API 服务。

### 5.2.3 SGLang 详解

SGLang 是 UC Berkeley 创业团队推出的 LLM 推理引擎，主打 Agent / 多轮对话场景。

核心创新：

RadixAttention：基于 Radix Tree 的前缀复用，支持部分前缀匹配，命中率比 vLLM 的 Prefix Cache 高 1.5-2 倍。

结构化生成：原生支持 JSON / Regex 约束输出，Function Calling 加速。

编程模型：提供 SGLang DSL，让用户用 Python 描述复杂推理流程。

典型用法：

```python
@sglang.function
def multi_turn(s, question):
    s += sglang.system("You are a helpful assistant.")
    s += sglang.user(question)
    s += sglang.assistant(sglang.gen("answer", max_tokens=256))
```

优势：

Agent / 多轮对话场景性能最强（前缀命中率高）。

结构化输出友好（Function Calling 速度提升 3-5 倍）。

编程模型直观。

劣势：

生态成熟度不如 vLLM。

通用场景（无前缀复用）优势不明显。

适用场景：AI Agent、多轮对话、RAG 系统、Function Calling 重度场景。

### 5.2.4 TGI 详解

TGI（Text Generation Inference） 是 HuggingFace 推出的 LLM 推理服务，主打易用性。

核心特性：

HuggingFace 生态深度集成：模型即用（telescope 模型库 100% 支持）。

OpenAI 兼容 API。

Continuous Batching + PagedAttention。

量化支持：GPTQ / AWQ / BitsAndBytes 全支持。

典型部署：

```bash
docker run -p 8080:80 ghcr.io/huggingface/text-generation-inference:latest \
  --model-id meta-llama/Llama-3-8B-Instruct
```

优势：

HuggingFace 生态无缝（Hub 模型一键部署）。

易用性极佳（Docker 一行启动）。

量化模型支持全面。

劣势：

性能不如 vLLM/TRT-LLM（通常慢 30-50%）。

多卡并行支持有限。

适用场景：HuggingFace 生态用户、快速原型、企业内部小规模部署。

### 5.2.5 MoE 模型框架支持度对比（新增）

MoE（Mixture of Experts）模型（如 DeepSeek-V3、Mixtral 8x7B）的推理与稠密模型有本质区别，需要框架提供专门的 MoE 支持。

主流框架对 MoE 模型的支持度对比：

| 框架 | MoE 推理 | 专家并行 | 负载均衡 | DeepSeek-V3 | Mixtral | 备注 |
| --- | --- | --- | --- | --- | --- | --- |
| vLLM | 完整 | 支持 | 支持 | 支持 | 支持 | 社区活跃，支持最全 |
| SGLang | 完整 | 支持 | 支持 | 支持 | 支持 | RadixAttention 优化 MoE |
| TRT-LLM | 完整 | 支持 | 支持 | 部分 | 支持 | 需在 MoE 层手工配置 |
| TGI | 部分 | 不支持 | 部分 | 部分 | 支持 | 对 DeepSeek 支持有限 |
| llama.cpp | 完整 | 不支持 | 部分 | 部分 | 支持 | 单机推理，不支持分布式 |
| MindIE | 完整 | 支持 | 支持 | 支持 | 支持 | 华为昇腾 MoE 方案 |

MoE 推理的关键优化：

专家并行（Expert Parallelism）：将不同 Expert 分配到不同 GPU，每步 All-to-All 通信

负载均衡路由：确保每个 Expert 的请求量均匀，避免热点 Expert 成为瓶颈

辅助损失优化：训练时加入负载均衡损失，从源头减少 Expert 不均衡

### 5.2.6 Agent 场景下推理框架选型（新增）

Agent 场景（如 Code Interpreter、Function Calling、Multi-Agent 协作）对推理框架提出了不同于聊天场景的独特要求。

Agent 场景 vs 聊天场景的推理需求差异：

| 维度 | 聊天场景 | Agent 场景 | Agent 的额外要求 |
| --- | --- | --- | --- |
| 请求特点 | 单轮对话 | 多轮 + 工具调用 | 需要维护工具调用状态 |
| 上下文长度 | 2K-32K | 32K-200K | 长上下文 + 多轮记忆 |
| 输出格式 | 自由文本 | 结构化 JSON + 代码 | 约束解码（Guided Decoding） |
| 并发模式 | 稳态并发 | 突发并发 + 串行依赖 | 动态调度 + 优先级队列 |
| 批处理效率 | Continuous Batching | Prefix Caching 更关键 | 系统 Prompt 复用 + 工具描述缓存 |

Agent 场景下的框架选型建议：

vLLM：适合通用 Agent 场景，支持 Function Calling 和约束解码，社区生态最丰富

SGLang：适合高并发 Agent 场景，RadixAttention 的 Prefix Caching 在 Agent 场景下效果显著

llama.cpp：适合端侧 Agent 场景（手机/PC 本地 Agent），资源占用低

## 5.3 端侧推理框架对比

### 5.3.1 llama.cpp 详解

llama.cpp 是社区最活跃的端侧 LLM 推理框架，由 Georgi Gerganov 2023 年开源。

核心特性：

纯 C++ 实现：无任何依赖，可在任何平台编译。

GGUF 格式：llama.cpp 专用模型格式，支持 Q2-Q8 多种量化。

量化生态完善：Q4_K_M / Q5_K_M / Q8_0 等多种精度档位。

CPU + GPU 混合：自动平衡 CPU/MPS/CUDA 计算。

OpenAI 兼容 API：自带 server 模式。

典型用法：

```bash
# 编译
make GGML_CUDA=1

# 量化
./quantize model-fp16.gguf model-q4_k_m.gguf q4_k_m

# 推理
./main -m model-q4_k_m.gguf -p "你好"
```

优势：

真正的跨平台（Linux/Mac/Windows/Android/iOS）。

内存占用极低（4GB 内存可跑 7B 模型）。

量化等级丰富（Q2-Q8 全覆盖）。

社区活跃，新模型第一时间支持。

劣势：

性能不如专用框架（如 Apple MLX 在 M3 上）。

多模态支持有限。

适用场景：本地开发调试、跨平台原型、消费级硬件部署、AR 眼镜/树莓派等资源受限设备。

### 5.3.2 MLC-LLM 详解

MLC-LLM 是 TVM 团队推出的跨平台 LLM 编译框架。

核心特性：

基于 TVM 编译：把 LLM 编译成各平台 native code（iOS Metal/Android Vulkan/WebGPU/Windows DX12）。

真正的跨平台：一份代码跑 iOS/Android/Web/Mac/Windows/Linux。

JSON 量化：支持 INT4/INT8 量化。

WebGPU 支持：浏览器内运行 LLM（如 web-llm 项目）。

典型用法：

```python
from mlc_llm import MLCEngine
engine = MLCEngine(model="Llama-3-8B-Instruct-q4f16_1-MLC")
response = engine.chat.completions.create(...)
```

优势：

跨平台最彻底（含 Web）。

性能优化深度高（TVM 的算子优化能力）。

移动端性能优于 llama.cpp（部分场景）。

劣势：

编译流程复杂（需 TVM 经验）。

社区相对小众。

模型支持较慢（需手动编译）。

适用场景：跨平台生产部署（同一份模型跑 iOS/Android/Web）、Web LLM 应用。

### 5.3.3 ExecuTorch 详解

ExecuTorch 是 Meta（PyTorch 团队）2023 年推出的端侧推理框架，是 PyTorch 官方端侧方案。

核心特性：

PyTorch 官方血脉：API 与 PyTorch eager 模式接近，学习成本低。

算子覆盖广：CNN / Transformer / LLM 全部支持。

多硬件后端：高通 HTP / Apple CoreML / ARM Ethos / MediaTek APU。

AOT 编译：通过 torch.export 提前编译。

典型用法：

```python
from executorch.pytorch import export
edge_program = export(model, (sample_input,))
edge_program.save("model.pte")
```

优势：

PyTorch 官方维护，长期演进有保障。

算子覆盖广（不止 LLM）。

多硬件支持（尤其高通 HTP 深度优化）。

劣势：

仍在快速迭代（API 可能变化）。

LLM 性能不如 llama.cpp（部分场景）。

文档相对薄弱。

适用场景：Meta 生态用户、需要同时部署 CNN + LLM、深度合作高通/苹果的项目。

### 5.3.5 端侧多模态推理框架

端侧多模态推理是 2025-2026 年最热门的推理方向。随着 Apple Intelligence、华为小艺、三星 Galaxy AI 等端侧 AI 的普及，在手机/平板/PC 上运行多模态模型成为刚需。

主流端侧多模态推理框架对比：

| 框架 | 支持平台 | 多模态类型 | 量化支持 | 推理速度 | 内存占用 |
| --- | --- | --- | --- | --- | --- |
| MLC-LLM | iOS/Android/Mac | 图文 | INT4/INT8 | 3-5 token/s | 2-4GB |
| llama.cpp | macOS/Linux/Win | 图文 | INT4/INT8/FP16 | 5-10 token/s | 1-3GB |
| ExecuTorch | iOS/Android | 图文+音频 | INT8 | 2-4 token/s | 1-2GB |
| MediaPipe | Android/iOS | 图文 | INT8 | 1-3 token/s | 500MB-1GB |
| CoreML | iOS/macOS | 图文+音频 | FP16/INT8 | 5-8 token/s | 1-3GB |

端侧多模态推理的关键技术：

视觉 Token 压缩：将 256-576 个视觉 Token 压缩到 32-64 个，减少端侧计算量

NPU 推理加速：Apple Neural Engine / 高通 Hexagon NPU 专为视觉计算优化

渐进式加载：先加载轻量级 Encoder，按需加载 LLM 权重，减少启动时间

## 5.4 芯片厂商 SDK 对比

### 5.4.1 高通 QNN/SNPE 详解

QNN（Qualcomm Neural Network） 是高通新一代 AI 推理 SDK，SNPE（Snapdragon Neural Processing Engine） 是旧版本。

核心特性：

多后端支持：HTP（Hexagon Tensor Processor，最强）/ GPU / CPU。

量化能力强：INT8/INT16/INT4 量化，HTP 16-bit 表现最佳。

多操作系统：Android / Windows on Snapdragon / Linux。

AI Hub：高通模型仓库，主流模型一键部署。

典型工作流：PyTorch → ONNX → QNN Converter → QNN Model Lib → qnn-net-run 推理

优势：

骁龙 8 Gen3 HTP 性能强劲（70B LLM 量化后可在手机跑）。

工业级稳定性（车规 8295 也用同一套）。

完善的工具链（converter/quantizer/profiler）。

劣势：

生态封闭（需注册 Qualcomm 账号 + NDA 才能下载 SDK）。

算子覆盖度有限（高级算子需 Custom Op）。

文档对外不公开。

适用场景：Android 手机端 AI、Windows on Snapdragon PC、车载 8295 座舱。

### 5.4.2 地平线 BPU SDK 详解

BPU SDK（伯努利架构）是地平线车规级 NPU 的官方 SDK，工具链名为"天工开物"。

核心特性：

hb_mapper：核心转换工具（PyTorch/ONNX → .bin 模型）。

量化能力强：支持 INT8/INT16，校准算法成熟。

车规认证：通过 ISO 26262 ASIL-B/D。

上板调试：完整 hbutil 工具集（hb_runtime/hb_benchmark/hb_view）。

典型工作流：

PyTorch → ONNX → hb_mapper converter → .bin       → hb_mapper quantizer（校准）→ 上板验证

优势：

车规级稳定（J3/J5/J6 已量产百万台）。

国产性价比高（比亚迪/大众主力供应商）。

工具链完善。

劣势：

仅支持地平线芯片。

算子覆盖度需查表（部分 LLM 算子不支持，需手写）。

LLM 支持有限（主要服务 CNN）。

适用场景：车载 ADAS / 智能座舱（J3/J5/J6/J6P）、国产机器人。

### 5.4.3 华为 CANN 详解

CANN（Compute Architecture for Neural Networks） 是华为昇腾 NPU 的官方 SDK。

核心特性：

ATC（Ascend Tensor Compiler）：核心转换工具（PyTorch/ONNX/TensorFlow → .om 模型）。

AscendCL：C++ 推理 API（类似 CUDA）。

MindIE：LLM 推理框架（昇腾版 vLLM）。

多芯片覆盖：昇腾 310（端侧）/ 910（云端）/ 910B（最新）。

典型工作流：PyTorch → ONNX → ATC → .om → AscendCL 推理

优势：

国产算力首选（信创项目必选）。

性能接近 NVIDIA A100（910B 实测）。

LLM 支持完善（MindIE 框架）。

劣势：

生态相对封闭（需华为账号 + 实名认证）。

算子覆盖度需查询（部分新算子滞后 NVIDIA）。

调试工具不如 NVIDIA（Nsight）成熟。

适用场景：信创/国产替代项目、政企客户、华为云用户。

### 5.4.4 瑞芯微 RKNN 详解

RKNN（Rockchip Neural Network） 是瑞芯微的端侧 NPU SDK，主打消费电子和IoT。

核心特性：

rknn-toolkit2：Python API（PC 上建模 + 量化）。

rknn-toolkit-lite2：C++ API（板端推理）。

多芯片支持：RK3566/3568/3588/3576。

算子覆盖广：CNN 主流算子全支持，Transformer 部分支持。

典型工作流：

```python
from rknn.api import RKNN
rknn = RKNN()
rknn.load_onnx(model='model.onnx')
rknn.build(do_quantization=True, dataset='calib_imgs/')
rknn.export_rknn('model.rknn')
```

优势：

芯片成本极低（RK3588 8 核 NPU 仅几百元）。

适合大规模 IoT 部署。

工具链简单易上手。

劣势：

NPU 算力有限（6 TOPS，跑大模型吃力）。

LLM 支持有限（主要服务 CNN）。

算子需查询支持表。

适用场景：消费电子（智能音箱/玩具/教育机器人）、IoT 设备、智慧屏。
